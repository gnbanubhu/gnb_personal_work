HOW TO BUILD LLMs (Large Language Models)?
===========================================

OVERVIEW
---------
Building an LLM from scratch is an enormous engineering and research undertaking.
This guide covers the full pipeline — from data collection to deployment — along
with practical alternatives like fine-tuning and RAG for teams that cannot train
from scratch.


LLM BUILDING PIPELINE (END-TO-END)
------------------------------------

  Step 1  →  Define Goals & Architecture
  Step 2  →  Data Collection & Curation
  Step 3  →  Tokenization
  Step 4  →  Model Architecture Design
  Step 5  →  Pre-Training
  Step 6  →  Supervised Fine-Tuning (SFT)
  Step 7  →  Reinforcement Learning from Human Feedback (RLHF)
  Step 8  →  Evaluation
  Step 9  →  Inference Optimization
  Step 10 →  Deployment & Monitoring


STEP 1 — DEFINE GOALS & ARCHITECTURE
--------------------------------------
  Before writing any code, answer:
    • What is the target use case? (general-purpose, code, domain-specific)
    • What context length is needed? (4K, 32K, 128K tokens)
    • What parameter scale? (7B, 13B, 70B, 405B)
    • What compute budget? (GPUs, TPUs, cloud spend)
    • Open-source or proprietary?

  Parameter scale vs capability:
    < 1B    → Simple classification, basic Q&A
    1B–7B   → Good instruction following, edge deployment
    7B–13B  → Strong reasoning, code, fits on 1-2 GPUs
    13B–70B → Near GPT-3.5 quality, multi-task excellence
    70B+    → GPT-4 class quality, requires multi-GPU cluster


STEP 2 — DATA COLLECTION & CURATION
--------------------------------------
  Data is the most critical ingredient. Quality > Quantity.

  Data Sources:
    • Web crawl data   : Common Crawl, C4, RefinedWeb
    • Books            : Books3, BookCorpus, Gutenberg
    • Code             : GitHub, Stack Overflow, The Stack
    • Scientific papers: arXiv, PubMed, Semantic Scholar
    • Wikipedia        : High-quality encyclopedic text
    • Curated datasets : The Pile, RedPajama, Dolma, SlimPajama

  Data Curation Pipeline:
    1. Deduplication   — remove exact and near-duplicate documents
    2. Language filter — keep target language(s)
    3. Quality filter  — remove boilerplate, spam, low-quality text
    4. Content filter  — remove harmful, private, or toxic content
    5. Domain mix      — balance web, books, code, science proportions
    6. Tokenization    — convert text to token IDs

  Typical pre-training data scale:
    7B  model  →  1–2 trillion tokens
    70B model  →  2–4 trillion tokens
    405B model →  15+ trillion tokens


STEP 3 — TOKENIZATION
-----------------------
  Tokenizers convert raw text into integer token IDs that the model processes.

  Common algorithms:
    BPE  (Byte Pair Encoding)    — GPT-2, GPT-4, LLaMA, Mistral
    WordPiece                    — BERT, DistilBERT
    SentencePiece (Unigram)      — T5, Gemma, PaLM

  Vocabulary size:
    Most modern LLMs use 32K–128K vocabulary size.
    LLaMA 3: 128K vocab. GPT-4: ~100K vocab.

  Building a tokenizer:
    from tokenizers import ByteLevelBPETokenizer
    tokenizer = ByteLevelBPETokenizer()
    tokenizer.train(files=["corpus.txt"], vocab_size=32000, min_frequency=2)
    tokenizer.save_model("tokenizer/")

  Key tokenizer properties:
    • Unknown tokens handled as byte fallback
    • Special tokens: <BOS>, <EOS>, <PAD>, <UNK>
    • Reversible: tokens → text without loss


STEP 4 — MODEL ARCHITECTURE DESIGN
-------------------------------------
  All modern LLMs are based on the Transformer architecture.
  Most use a Decoder-Only design (autoregressive generation).

  Core Transformer components:
    Input Embeddings        Convert token IDs to dense vectors
    Positional Encoding     Encode position in sequence (RoPE, ALiBi)
    Multi-Head Attention    Attend to all previous tokens simultaneously
    Feed-Forward Network    2-layer MLP per token (often with SwiGLU)
    Layer Normalization     Stabilize training (RMSNorm preferred)
    Output Head             Linear layer projecting to vocabulary size

  Modern architecture improvements:
    RoPE Positional Encoding   Rotary position — better long context
    GQA (Grouped Query Attention)  Fewer KV heads — reduces memory
    SwiGLU Activation          Better than ReLU/GELU in feed-forward
    RMSNorm                    Faster and stabler than LayerNorm
    Flash Attention            Memory-efficient attention computation
    Sliding Window Attention   Handle very long contexts (Mistral)

  Example architecture (LLaMA-style 7B):
    Layers              : 32
    Hidden dimension    : 4096
    Attention heads     : 32
    KV heads (GQA)      : 8
    Feed-forward dim    : 11008
    Context length      : 8192 tokens
    Vocabulary          : 128K
    Total parameters    : ~7 billion


STEP 5 — PRE-TRAINING
-----------------------
  The model is trained to predict the next token given all previous tokens.
  This is called Causal Language Modeling (CLM).

  Objective:
    Minimize cross-entropy loss:
    Loss = -Σ log P(token_t | token_1, ..., token_{t-1})

  Training infrastructure:
    Hardware   : A100/H100 GPUs or TPU v4/v5 pods
    Parallelism: Data + Tensor + Pipeline parallelism
    Frameworks : Megatron-LM, DeepSpeed, PyTorch FSDP, JAX/XLA

  Key hyperparameters:
    Learning rate     : 1e-4 to 3e-4 with cosine decay
    Batch size        : 1M–4M tokens per step (packed sequences)
    Optimizer         : AdamW (β1=0.9, β2=0.95, weight_decay=0.1)
    Gradient clipping : 1.0
    Warmup steps      : 2000–4000 steps
    Precision         : BF16 mixed precision

  Compute requirements:
    7B  model, 1T tokens  →  ~100K A100-hours  (~$200K–$500K)
    70B model, 2T tokens  →  ~1M A100-hours    (~$2M–$5M)
    405B model            →  ~10M+ GPU-hours   (~$20M–$50M+)

  Training frameworks:
    Megatron-LM   — NVIDIA, handles 100B+ model training
    DeepSpeed     — Microsoft, ZeRO optimization for memory
    PyTorch FSDP  — Fully Sharded Data Parallel, native PyTorch
    JAX + XLA     — Google, TPU-optimized training


STEP 6 — SUPERVISED FINE-TUNING (SFT)
----------------------------------------
  After pre-training, the base model only predicts text — it does not follow
  instructions. SFT teaches the model to respond to prompts helpfully.

  Training data format:
    {"instruction": "Summarize this article.",
     "input":       "..article text..",
     "output":      "..summary.."}

  Popular SFT datasets:
    Alpaca       — 52K GPT-4 generated instruction pairs
    ShareGPT     — real user ChatGPT conversations
    OpenHermes   — 1M high-quality instruction pairs
    UltraChat    — 1.5M multi-turn dialogue pairs

  Efficient fine-tuning methods:
    Full Fine-Tuning   — update all parameters (expensive)
    LoRA               — Low-Rank Adaptation, train small adapter matrices
    QLoRA              — LoRA on quantized model (4-bit), fits on 1 GPU
    Prefix Tuning      — prepend learned soft tokens to input
    Adapter Layers     — insert small trainable modules between layers

  LoRA example with Hugging Face:
    from peft import LoraConfig, get_peft_model
    config = LoraConfig(r=16, lora_alpha=32,
                        target_modules=["q_proj","v_proj"],
                        lora_dropout=0.05)
    model  = get_peft_model(base_model, config)

  SFT training tools:
    Hugging Face TRL    — SFTTrainer, easy fine-tuning
    Axolotl             — flexible fine-tuning framework
    LLaMA-Factory       — supports many models and methods
    Unsloth             — 2× faster LoRA fine-tuning


STEP 7 — RLHF (Reinforcement Learning from Human Feedback)
------------------------------------------------------------
  RLHF aligns the model to be helpful, harmless, and honest by optimizing
  for human preferences rather than just next-token prediction.

  RLHF Pipeline:
    1. Collect human preference data
         Show raters pairs of model responses → rater picks the better one
    2. Train a Reward Model (RM)
         Binary classifier trained on preference data to score responses
    3. PPO Training (Proximal Policy Optimization)
         Fine-tune the LLM using the reward model as a signal
         Maximize reward while staying close to the SFT model (KL penalty)

  Alternatives to PPO:
    DPO  (Direct Preference Optimization)  — simpler, no separate reward model
    RLAIF (RL from AI Feedback)            — use another LLM as the rater
    KTO  (Kahneman-Tversky Optimization)   — works with binary feedback

  DPO example (Hugging Face TRL):
    from trl import DPOTrainer
    trainer = DPOTrainer(model, ref_model, args=training_args,
                         beta=0.1, train_dataset=preference_dataset)
    trainer.train()


STEP 8 — EVALUATION
---------------------
  Evaluate the model across multiple dimensions before release.

  Benchmark categories:
    Reasoning      : MMLU, HellaSwag, WinoGrande, ARC
    Code           : HumanEval, MBPP, SWE-bench
    Math           : GSM8K, MATH, AIME
    Language       : LAMBADA, WMT translation benchmarks
    Safety         : TruthfulQA, BBQ, WinoBias
    Long context   : RULER, SCROLLS, LongBench
    Instruction    : MT-Bench, AlpacaEval, Chatbot Arena

  Evaluation tools:
    LM Evaluation Harness  — standard benchmark runner (EleutherAI)
    HELM                   — holistic evaluation framework (Stanford)
    OpenCompass            — multi-model, multi-benchmark evaluation
    Chatbot Arena          — human preference ELO ranking


STEP 9 — INFERENCE OPTIMIZATION
---------------------------------
  Trained models need to be fast and memory-efficient for production.

  Quantization (reduce model size):
    FP32   → 4 bytes/param  (training precision)
    BF16   → 2 bytes/param  (standard inference)
    INT8   → 1 byte/param   (~0.5% quality loss)
    INT4   → 0.5 bytes/param (GGUF, GPTQ, AWQ formats)

    Tools: bitsandbytes, AutoGPTQ, llama.cpp, Ollama

  Attention optimization:
    KV Cache          — cache key/value pairs to avoid recomputation
    Flash Attention 2 — memory-efficient attention (10× speedup)
    Paged Attention   — vLLM's memory management for KV cache

  Serving frameworks:
    vLLM              — high-throughput LLM serving, continuous batching
    TensorRT-LLM      — NVIDIA optimized inference engine
    llama.cpp         — CPU/GPU inference in C++, supports GGUF
    Ollama            — local LLM runner for Mac/Linux/Windows
    Hugging Face TGI  — Text Generation Inference server


STEP 10 — DEPLOYMENT & MONITORING
-----------------------------------
  Deployment options:
    Cloud API         — OpenAI, Anthropic, Together AI, Groq
    Self-hosted GPU   — vLLM / TGI on A10G/A100/H100
    Edge / Local      — llama.cpp, Ollama on CPU/Apple Silicon
    Serverless        — AWS Bedrock, Azure OpenAI, Google Vertex AI

  Monitoring metrics to track:
    Latency           — Time to First Token (TTFT), tokens/second
    Throughput        — requests per second, tokens per second
    Cost              — USD per 1K tokens, cost per request
    Quality           — user ratings, task success rate
    Safety            — harmful output rate, refusal rate
    Drift             — output quality degradation over time


PRACTICAL PATHS — YOU DON'T HAVE TO BUILD FROM SCRATCH
--------------------------------------------------------

  Path 1 — Use a Cloud LLM API                  (easiest)
    When: You need fast results with no infra
    How:  Call Anthropic, OpenAI, or Google APIs
    Cost: Pay per token

  Path 2 — Fine-tune an Open-Source Model        (practical)
    When: You need domain adaptation or custom behavior
    How:  QLoRA fine-tune LLaMA 3 / Mistral on your data
    Cost: 1-2 GPUs for hours to days

  Path 3 — RAG (Retrieval-Augmented Generation)  (recommended for knowledge)
    When: You want the model to answer from YOUR documents
    How:  Embed documents → FAISS index → retrieve + generate
    Cost: Embedding compute + LLM API calls

  Path 4 — Pre-train from Scratch                (rare)
    When: You have unique data, scale, and a research team
    How:  Megatron-LM + thousands of GPUs + months of training
    Cost: Millions of dollars

SUMMARY
--------
Building an LLM requires:
  1.  Defining architecture and goals
  2.  Collecting and curating trillions of tokens of clean text
  3.  Training a tokenizer (BPE / SentencePiece)
  4.  Designing a Transformer model (decoder-only, RoPE, GQA, SwiGLU)
  5.  Pre-training with causal language modeling at massive scale
  6.  Supervised fine-tuning (SFT) for instruction following
  7.  RLHF or DPO for alignment with human preferences
  8.  Rigorous evaluation across benchmarks
  9.  Inference optimization (quantization, Flash Attention, vLLM)
  10. Deployment with monitoring for latency, cost, and quality

For most teams, fine-tuning or RAG on top of an existing open-source
model (LLaMA, Mistral, Gemma) is the right approach — far more practical
than training from scratch.
