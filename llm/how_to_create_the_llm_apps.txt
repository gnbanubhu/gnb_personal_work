HOW TO CREATE LLM APPLICATIONS?
=================================

OVERVIEW
---------
Building an LLM application means combining a language model with your own
data, logic, and interfaces to solve a real business problem. You do NOT need
to train an LLM — you use an existing one via API and build the application
layer around it.

LLM Application = LLM (brain) + Your Data + Your Logic + Your Interface


LLM APPLICATION ARCHITECTURE
------------------------------

  ┌─────────────────────────────────────────────────────────────────┐
  │                    LLM APPLICATION STACK                        │
  │                                                                 │
  │  User Interface   (Chat UI, API, CLI, Web App)                  │
  │        ↓                                                        │
  │  Application Layer (Orchestration, Routing, Memory)            │
  │        ↓                                                        │
  │  LLM Layer        (Claude, GPT-4, LLaMA, Mistral)              │
  │        ↓                                                        │
  │  Data Layer       (Vector DB, SQL DB, Files, APIs)             │
  └─────────────────────────────────────────────────────────────────┘


TYPES OF LLM APPLICATIONS
---------------------------

  1. Chatbot / Conversational Assistant
     Multi-turn dialogue with memory and context
     Examples: Customer support bot, HR assistant, data engineering helper

  2. RAG Application (Retrieval-Augmented Generation)
     Answer questions grounded in your own documents
     Examples: Internal wiki Q&A, policy assistant, document search

  3. Agentic Application
     LLM autonomously decides which tools to call to complete a task
     Examples: Data pipeline debugger, code agent, research agent

  4. Text Processing Pipeline
     Batch process documents: classify, extract, summarize, translate
     Examples: Invoice extraction, log classification, report summarization

  5. Code Generation App
     Generate, explain, review, or refactor code from natural language
     Examples: SQL generator, PySpark assistant, test generator

  6. Semantic Search Engine
     Find documents by meaning rather than keywords
     Examples: Knowledge base search, product search, support ticket routing


STEP-BY-STEP GUIDE TO BUILDING AN LLM APPLICATION
----------------------------------------------------

STEP 1 — DEFINE THE USE CASE
  Ask:
    • What problem does this solve?
    • Who are the users?
    • What is the input? (text, documents, structured data)
    • What is the expected output? (answer, JSON, code, summary)
    • What data does the LLM need to answer correctly?

  Example:
    Problem   : Engineers waste time searching internal Confluence docs
    Input     : User question in natural language
    Output    : Grounded answer with source references
    Data      : Confluence pages embedded in a vector store


STEP 2 — CHOOSE YOUR LLM
  Criteria              Options
  ──────────────────────────────────────────────────────────────────
  Best quality          Claude Opus, GPT-4o, Gemini Ultra
  Balanced              Claude Sonnet, GPT-4o-mini, Gemini Flash
  Low cost / fast       Claude Haiku, GPT-3.5, Mistral 7B
  On-premise / private  LLaMA 3, Mistral, Gemma (via Ollama/vLLM)
  Code-specialized      CodeLlama, Codestral, DeepSeek Coder
  Multilingual          Claude, GPT-4o, Gemini (all strong)

  Rule of thumb:
    Route simple tasks  → Haiku / GPT-3.5  (cheap, fast)
    Route complex tasks → Sonnet / GPT-4o  (balanced)
    Route hardest tasks → Opus / GPT-4o    (best quality)


STEP 3 — DESIGN YOUR PROMPT
  The prompt is the most important part of any LLM application.

  Prompt structure:
    ┌──────────────────────────────────────────┐
    │ System Prompt  (role, rules, format)     │
    │ Context        (retrieved docs, data)    │
    │ Examples       (few-shot if needed)      │
    │ User Input     (the actual question)     │
    └──────────────────────────────────────────┘

  Best practices:
    • Be specific about the role: "You are a senior data engineer..."
    • Define the output format: "Respond in JSON", "Use bullet points"
    • Set constraints: "Answer only from the context provided"
    • Use few-shot examples for custom formats or domain tasks
    • Add negative instructions: "Do not make up data"

  Example system prompt for a data engineering assistant:
    "You are a data engineering assistant specializing in Apache Spark,
     SQL, and cloud data platforms. Answer only using the provided context.
     If the answer is not in the context, say 'I don't know.'
     Always cite the source document in your answer."


STEP 4 — CONNECT YOUR DATA (RAG Pattern)
  Most LLM apps need to ground answers in your own data.

  RAG pipeline:
    a) Ingest documents
         Load PDFs, HTML, Markdown, CSV, database records

    b) Chunk documents
         Split into 200–500 word chunks with optional overlap

    c) Embed chunks
         Use an embedding model to convert text to vectors
         Models: all-MiniLM-L6-v2, text-embedding-3-small (OpenAI)

    d) Store in vector database
         FAISS (local), Pinecone, Weaviate, Chroma, pgvector

    e) At query time:
         Embed the user question
         Retrieve top-K most similar chunks
         Inject chunks into the LLM prompt as context

  Code example (Python):
    from sentence_transformers import SentenceTransformer
    model   = SentenceTransformer("all-MiniLM-L6-v2")
    chunks  = ["doc chunk 1", "doc chunk 2", ...]
    vectors = model.encode(chunks, normalize_embeddings=True)

    # Store in FAISS
    import faiss, numpy as np
    index = faiss.IndexFlatIP(384)
    index.add(vectors.astype("float32"))

    # Retrieve for a query
    q_vec = model.encode(["user question"], normalize_embeddings=True)
    scores, indices = index.search(q_vec.astype("float32"), k=5)


STEP 5 — BUILD THE ORCHESTRATION LAYER
  The orchestration layer manages:
    • Multi-turn conversation history
    • Context window management (summarize old history)
    • Tool/function calling
    • Routing queries to the right LLM or data source
    • Error handling and retries

  Orchestration frameworks:
    LangChain     — most popular, chains + agents + RAG pipelines
    LlamaIndex    — specialized for document indexing and RAG
    Haystack      — production RAG and NLP pipelines
    Semantic Kernel — Microsoft, .NET + Python LLM orchestration
    DSPy          — programmatic LLM pipeline optimization
    Custom Python — often simplest for focused applications

  Conversation memory patterns:
    Buffer Memory       — store last N messages verbatim
    Summary Memory      — summarize older messages to save tokens
    Vector Memory       — embed and retrieve relevant past messages
    Entity Memory       — track key entities across the conversation


STEP 6 — IMPLEMENT TOOL USE (AGENTS)
  Give the LLM tools it can call to fetch data or perform actions.

  Tool use pattern:
    1. Define tools with name, description, and input schema
    2. Send tools + user query to the LLM
    3. LLM returns a tool_use block (name + arguments)
    4. Your code executes the tool and returns results
    5. LLM receives results and generates final answer
    6. Repeat until stop_reason = "end_turn"

  Example tools for a data engineering agent:
    get_pipeline_status(pipeline_name)
    run_sql_query(query, database)
    get_table_schema(table_name)
    check_data_quality(dataset_name)
    trigger_pipeline_run(pipeline_name)

  Code pattern (Anthropic SDK):
    while True:
        response = client.messages.create(
            model=MODEL, tools=TOOLS, messages=messages
        )
        if response.stop_reason == "tool_use":
            # execute tools, append results to messages
        elif response.stop_reason == "end_turn":
            return response.content[0].text


STEP 7 — HANDLE OUTPUT PARSING
  LLM outputs are text — you often need structured data.

  Techniques:
    a) Instruct JSON output
         Prompt: "Respond with ONLY valid JSON. Schema: {...}"

    b) Output parsers
         Parse JSON, markdown tables, bullet lists from response

    c) Structured output (function calling)
         Define a response schema as a tool — model fills it in

    d) Retry on parse failure
         If JSON parse fails, retry with "Fix this JSON: {bad_output}"

  Example JSON extraction:
    import json, anthropic
    response = client.messages.create(
        model="claude-opus-4-6", max_tokens=200,
        messages=[{"role": "user",
                   "content": "Extract name, role, company as JSON from: "
                              "'John Smith is a Data Engineer at Acme Corp'"}]
    )
    data = json.loads(response.content[0].text)


STEP 8 — ADD SAFETY & GUARDRAILS
  Prevent the application from producing harmful or incorrect outputs.

  Input guardrails:
    • Block prompt injection attempts
    • Validate input length and format
    • Filter disallowed topics or intents

  Output guardrails:
    • Check response is grounded in context (faithfulness check)
    • Detect hallucinated facts or made-up references
    • Filter sensitive data (PII, credentials) from outputs
    • LLM-as-judge: use another LLM call to validate the response

  Tools:
    Guardrails AI       — define rules and validators for LLM outputs
    NeMo Guardrails     — NVIDIA, conversational safety rails
    Custom validators   — regex, schema check, semantic similarity check


STEP 9 — BUILD THE USER INTERFACE
  Options based on use case:

    REST API          — FastAPI / Flask endpoint for LLM calls
    Streaming API     — Server-Sent Events (SSE) for real-time output
    Chat UI           — Streamlit, Gradio, Chainlit for quick prototypes
    Slack / Teams bot — integrate via webhook for internal tools
    Web app           — React / Next.js frontend calling your LLM API
    CLI tool          — Python click / argparse for developer tools

  Streaming example (FastAPI + Anthropic):
    @app.post("/chat")
    async def chat(request: ChatRequest):
        async def generate():
            with client.messages.stream(...) as stream:
                for text in stream.text_stream:
                    yield f"data: {text}\n\n"
        return StreamingResponse(generate(), media_type="text/event-stream")


STEP 10 — TEST, EVALUATE & MONITOR
  Testing LLM apps is different from traditional unit tests.

  Testing approaches:
    Unit tests        — test prompt templates, parsers, tool logic
    Regression tests  — golden dataset of question → expected answer
    LLM-as-judge      — use GPT-4/Claude to rate response quality
    A/B testing       — compare prompt variants on real user traffic
    Red teaming       — adversarial testing for safety and robustness

  Evaluation metrics:
    Faithfulness      — is the answer grounded in retrieved context?
    Relevance         — does the answer address the question?
    Completeness      — are all parts of the question answered?
    Conciseness       — is the answer appropriately brief?
    Safety            — is the output free of harmful content?

  Monitoring in production:
    Log every request  — prompt, response, tokens, cost, latency
    Track cost         — USD per call, per user, per day
    Alert on failures  — errors, timeouts, budget thresholds
    User feedback      — thumbs up/down, ratings, corrections

  Monitoring tools:
    LangSmith          — LangChain's tracing and evaluation platform
    Weights & Biases   — experiment tracking + LLM monitoring
    Arize AI           — LLM observability and drift detection
    Helicone           — lightweight LLM proxy with logging
    Custom logging     — structured JSON logs → ELK / Datadog / Grafana


COMMON LLM APPLICATION PATTERNS
---------------------------------

  Pattern 1 — Simple Q&A
    User question → System prompt + question → LLM → Answer

  Pattern 2 — RAG (Document Q&A)
    User question → Embed → Retrieve chunks → Prompt + chunks → LLM → Answer

  Pattern 3 — Conversational RAG
    Chat history + User question → Retrieve → Prompt + history + chunks → Answer

  Pattern 4 — Single Agent
    User task → LLM decides tool → Execute tool → LLM uses result → Answer

  Pattern 5 — Multi-Agent
    Orchestrator LLM → delegates to specialist agents (coder, searcher, analyst)
    Each agent has its own tools and system prompt

  Pattern 6 — Map-Reduce
    Large document → Split into chunks → LLM processes each chunk (map)
    → LLM combines all summaries (reduce) → Final answer

  Pattern 7 — Chain of Thought Pipeline
    Step 1: Extract key facts → Step 2: Reason over facts → Step 3: Generate answer

QUICK START CODE — MINIMAL RAG APP
------------------------------------

  import os, anthropic, numpy as np
  from sentence_transformers import SentenceTransformer

  client     = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
  embedder   = SentenceTransformer("all-MiniLM-L6-v2")

  # 1. Index your documents
  docs       = ["Doc 1 content...", "Doc 2 content...", "Doc 3 content..."]
  doc_vecs   = embedder.encode(docs, normalize_embeddings=True)

  # 2. Retrieve relevant docs for a query
  def retrieve(query, top_k=3):
      q_vec  = embedder.encode([query], normalize_embeddings=True)[0]
      scores = doc_vecs @ q_vec
      top_idx = np.argsort(scores)[::-1][:top_k]
      return [docs[i] for i in top_idx]

  # 3. Generate grounded answer
  def ask(question):
      context = "\n\n".join(retrieve(question))
      response = client.messages.create(
          model="claude-opus-4-6", max_tokens=300,
          system="Answer only from the context. Cite sources.",
          messages=[{"role": "user",
                     "content": f"Context:\n{context}\n\nQuestion: {question}"}]
      )
      return response.content[0].text

  print(ask("What is Apache Spark used for?"))


SUMMARY
--------
Building an LLM application involves 10 steps:

  1.  Define the use case        — problem, input, output, data
  2.  Choose your LLM            — Haiku / Sonnet / Opus based on complexity
  3.  Design your prompt         — system prompt, context, format, constraints
  4.  Connect your data (RAG)    — embed, index, retrieve, inject context
  5.  Build orchestration        — conversation memory, routing, retries
  6.  Implement tool use         — agentic loop for autonomous task execution
  7.  Handle output parsing      — JSON, structured extraction, retry on failure
  8.  Add safety & guardrails    — input/output validation, faithfulness checks
  9.  Build the user interface   — API, chat UI, Slack bot, web app
  10. Test, evaluate & monitor   — regression tests, LLM-as-judge, cost tracking

Key principle:
  Start simple (direct API call) → add RAG → add tools → add agents.
  Do not over-engineer before validating the core use case works.
