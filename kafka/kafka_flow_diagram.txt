DIAGRAM 2: PRODUCER MESSAGE FLOW (Step-by-Step)
====================================================================

  APPLICATION
      │
      │  1. Creates event/record
      ▼
  ┌─────────────────────────────────────┐
  │         PRODUCER CLIENT             │
  │                                     │
  │  2. SERIALIZE                       │
  │     Key   → StringSerializer        │
  │     Value → JsonSerializer / Avro   │
  └──────────────┬──────────────────────┘
                 │
                 ▼
  ┌─────────────────────────────────────┐
  │         PARTITIONER                 │
  │                                     │
  │  3. DECIDE PARTITION                │
  │                                     │
  │  Key provided?                      │
  │    YES → hash(key) % num_partitions │
  │    NO  → Round-robin / Sticky       │
  └──────────────┬──────────────────────┘
                 │
                 ▼
  ┌─────────────────────────────────────┐
  │         RECORD ACCUMULATOR          │
  │         (In-memory buffer)          │
  │                                     │
  │  4. BATCH messages per partition    │
  │     Wait until:                     │
  │       batch.size reached  OR        │
  │       linger.ms elapsed             │
  └──────────────┬──────────────────────┘
                 │
                 ▼
  ┌─────────────────────────────────────┐
  │         COMPRESSION (optional)      │
  │  gzip / snappy / lz4 / zstd        │
  └──────────────┬──────────────────────┘
                 │
                 ▼
  ┌─────────────────────────────────────┐
  │         SEND TO KAFKA BROKER        │
  │         (Leader of partition)       │
  │                                     │
  │  5. ACKNOWLEDGMENT                  │
  │                                     │
  │  acks=0  → No wait (fastest)        │
  │            May lose messages        │
  │                                     │
  │  acks=1  → Wait for Leader only     │
  │            Moderate reliability     │
  │                                     │
  │  acks=all→ Wait for all ISR         │
  │            Strongest guarantee      │
  └──────────────┬──────────────────────┘
                 │
         ┌───────┴───────┐
         │               │
         ▼               ▼
      SUCCESS          FAILURE
         │               │
         ▼               ▼
  Record offset     Retry (up to
  returned to       retries config)
  application       └──► Dead Letter
                         Queue (if
                         retries exhausted)

DIAGRAM 4: CONSUMER GROUP FLOW & PARTITION ASSIGNMENT
====================================================================

  SCENARIO: Topic "orders" has 6 partitions

  ┌────────────────────────────────────────────────────────────────┐
  │                TOPIC: orders (6 partitions)                    │
  │   P0   P1   P2   P3   P4   P5                                 │
  └────┬────┬────┬────┬────┬────┬───────────────────────────────┘
       │    │    │    │    │    │
       └────┴────┴────┴────┴────┴──────────────┐
                                               │
                                               ▼
  ┌────────────────────────────────────────────────────────────────┐
  │            CONSUMER GROUP: "order-processor"                   │
  │                                                                │
  │  Consumer 1         Consumer 2         Consumer 3             │
  │  ┌──────────┐       ┌──────────┐       ┌──────────┐          │
  │  │ reads    │       │ reads    │       │ reads    │          │
  │  │ P0, P1   │       │ P2, P3   │       │ P4, P5   │          │
  │  └──────────┘       └──────────┘       └──────────┘          │
  └────────────────────────────────────────────────────────────────┘

  ── Consumer fails (Consumer 2 goes down) → REBALANCE ──

  ┌────────────────────────────────────────────────────────────────┐
  │            CONSUMER GROUP: "order-processor"  (after rebalance)│
  │                                                                │
  │  Consumer 1              Consumer 3                           │
  │  ┌──────────────┐        ┌──────────────┐                    │
  │  │ reads        │        │ reads        │                    │
  │  │ P0, P1, P2   │        │ P3, P4, P5   │                    │
  │  └──────────────┘        └──────────────┘                    │
  └────────────────────────────────────────────────────────────────┘

  ── Multiple Consumer Groups read independently ──

  P0..P5  ──────────────►  Group A: "order-processor"  (fulfillment)
          ──────────────►  Group B: "analytics-service" (reporting)
          ──────────────►  Group C: "fraud-detector"    (fraud check)

  Each group maintains its own offsets — fully independent reads

DIAGRAM 5: REPLICATION FLOW (Fault Tolerance)
====================================================================

  Topic: "payments", Partition 0
  Replication Factor = 3

  ┌──────────────────────────────────────────────────────────────┐
  │                      PRODUCER                                │
  └──────────────────────────────┬───────────────────────────────┘
                                 │  write
                                 ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  BROKER 1 — payments-P0 LEADER                               │
  │  Receives write, appends to local log                        │
  │                                                              │
  │  [msg0][msg1][msg2][msg3][msg4]◄── new write                 │
  └────────────────┬───────────────────────────────────────────┘
                   │  replicate to followers
       ┌───────────┴──────────┐
       │                      │
       ▼                      ▼
  ┌──────────────┐     ┌──────────────┐
  │  BROKER 2    │     │  BROKER 3    │
  │  FOLLOWER    │     │  FOLLOWER    │
  │  (ISR)       │     │  (ISR)       │
  │              │     │              │
  │ [msg0][msg1] │     │ [msg0][msg1] │
  │ [msg2][msg3] │     │ [msg2][msg3] │
  │ [msg4]       │     │ [msg4]       │
  └──────────────┘     └──────────────┘

  ── Broker 1 FAILS ──

  ┌──────────────────────────────────────────────────────────────┐
  │  KRaft Controller detects Leader failure                      │
  │  Elects new Leader from ISR (e.g., Broker 2)                 │
  └──────────────────────────────────────────────────────────────┘
                                 │
                                 ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  BROKER 2 — payments-P0 NEW LEADER                           │
  │  Producers and consumers automatically reconnect             │
  │  No data loss (all messages were replicated)                 │
  └──────────────────────────────────────────────────────────────┘

DIAGRAM 10: DELIVERY SEMANTICS FLOW
====================================================================

  ── AT-MOST-ONCE (may lose messages) ──

  Producer ──► Kafka ──► Consumer reads
                         Consumer commits offset BEFORE processing
                             │
                         Consumer crashes
                             │
                         Message LOST (offset already committed)

  ── AT-LEAST-ONCE (may duplicate — most common) ──

  Producer ──► Kafka ──► Consumer reads
                         Consumer processes message
                         Consumer commits offset AFTER processing
                             │
                         Consumer crashes BEFORE commit
                             │
                         Consumer restarts → reads SAME message again
                         Message DUPLICATED → consumer must be idempotent

  ── EXACTLY-ONCE (no loss, no duplicate) ──

  Producer                         Kafka
  ┌──────────────────┐            ┌──────────────────────────┐
  │ idempotent=true  │            │                          │
  │ producer_id=42   │            │ Detects duplicate by     │
  │ sequence_num=10  │──────────► │ producer_id + seq_num    │
  │                  │            │ Deduplicates automatically│
  └──────────────────┘            └──────────────────────────┘

  Consumer + Producer (Kafka Streams):
  ┌─────────────────────────────────────────────────────────┐
  │  BEGIN TRANSACTION                                      │
  │    Read from input topic (offset 47)                   │
  │    Process message                                      │
  │    Write result to output topic                        │
  │    Commit input offset atomically with output write    │
  │  COMMIT TRANSACTION                                     │
  │                                                         │
  │  → Either ALL happens or NONE happens                  │
  └─────────────────────────────────────────────────────────┘

DIAGRAM 12: LOG COMPACTION FLOW
====================================================================

  COMPACTED TOPIC: "inventory-state"
  (Keep only LATEST record per key)

  BEFORE COMPACTION:
  ┌────────────────────────────────────────────────────────────┐
  │  offset │  key    │  value                                 │
  │  ──────────────────────────────────────────────────────    │
  │    0    │ sku-001 │ { stock: 100 }                        │
  │    1    │ sku-002 │ { stock: 50  }                        │
  │    2    │ sku-001 │ { stock: 95  }  ← updated             │
  │    3    │ sku-003 │ { stock: 200 }                        │
  │    4    │ sku-002 │ { stock: 45  }  ← updated             │
  │    5    │ sku-001 │ { stock: 90  }  ← updated             │
  │    6    │ sku-002 │ null            ← deleted (tombstone) │
  └────────────────────────────────────────────────────────────┘
                             │
                   Log Compaction runs
                             │
                             ▼
  AFTER COMPACTION:
  ┌────────────────────────────────────────────────────────────┐
  │  offset │  key    │  value                                 │
  │  ──────────────────────────────────────────────────────    │
  │    5    │ sku-001 │ { stock: 90  }  ← latest kept         │
  │    3    │ sku-003 │ { stock: 200 }  ← only record, kept   │
  │         │ sku-002 │  (removed — tombstone purged)          │
  └────────────────────────────────────────────────────────────┘

  Use cases for compacted topics:
    - Database-like snapshots (latest state per key)
    - User profile store
    - Inventory state
    - Configuration store

SUMMARY OF ALL DIAGRAMS
====================================================================

  Diagram  1 : High-Level Kafka Architecture
               Producers → Kafka Cluster (brokers + KRaft) → Consumer Groups

  Diagram  2 : Producer Message Flow
               Create → Serialize → Partition → Batch → Compress → Send → ACK

  Diagram  3 : Topic Partition & Offset Flow
               Partitions with offsets, same key → same partition

  Diagram  4 : Consumer Group & Partition Assignment
               Balanced assignment, rebalance on failure, independent groups

  Diagram  5 : Replication Flow
               Leader + ISR followers, automatic leader election on failure

  Diagram  6 : End-to-End Order Processing Flow
               Order placed → Kafka → Inventory, Payment, Notification, Fraud

  Diagram  7 : Consumer Offset Commit Flow
               Poll → Deserialize → Process → Commit (auto or manual)

  Diagram  8 : Kafka Connect Flow
               Source: DB → Kafka (CDC), Sink: Kafka → S3/BigQuery/Elastic

  Diagram  9 : Kafka Streams Processing
               Filter → Map → GroupBy → Window → Aggregate → Output topic

  Diagram 10 : Delivery Semantics
               At-most-once, At-least-once, Exactly-once with transactions

  Diagram 11 : Kafka in Data Engineering Pipeline
               Sources → Kafka → Stream Processing / Connect → Dashboards / ML

  Diagram 12 : Log Compaction Flow
               Retain only latest record per key, tombstone for deletes
