APACHE KAFKA вҖ” CONCEPTS
========================

Apache Kafka is an open-source distributed event streaming platform
originally developed by LinkedIn and donated to the Apache Software
Foundation in 2011. It is designed to handle high-throughput, fault-tolerant,
real-time data streaming at massive scale вҖ” processing trillions of events
per day at companies like LinkedIn, Uber, Netflix, Airbnb, and Twitter.

------------------------------------------------------------
WHAT IS KAFKA?
------------------------------------------------------------

Kafka is fundamentally a distributed commit log вҖ” an ordered, immutable
sequence of records that can be read by multiple consumers independently.

  Traditional Message Queue:        Apache Kafka:

Key difference:
  - Traditional queues: message deleted after one consumer reads it
  - Kafka: message retained (days/weeks), multiple consumers read independently
  - Kafka is both a message queue AND a storage system AND a stream processor


------------------------------------------------------------
CORE KAFKA CONCEPTS
------------------------------------------------------------

1. EVENT (Message / Record)
   -------------------------
   The fundamental unit of data in Kafka.
   An event is an immutable record of something that happened.

   Structure of a Kafka Event:
   в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
   в”Ӯ  Key      : "user-123"  (optional)           в”Ӯ
   в”Ӯ  Value    : {"action": "purchase", ...}      в”Ӯ
   в”Ӯ  Timestamp: 2024-01-15T10:30:00Z             в”Ӯ
   в”Ӯ  Headers  : {"source": "web-app"}  (optional)в”Ӯ
   в”Ӯ  Offset   : 4523  (position in partition)    в”Ӯ
   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

   - Key   : Used for partitioning (same key вҶ’ same partition)
   - Value : The actual payload (JSON, Avro, Protobuf, String)
   - Offset: Unique ID within a partition (monotonically increasing)


2. TOPIC
   ------
   A topic is a named channel to which events are written and from which
   events are read. Think of it like a folder or a database table.

   Examples of topics:
     - orders
     - payments
     - user-clicks
     - inventory-updates
     - shipment-events

   Properties:
     - Topics are append-only вҖ” producers only add to the end
     - Topics are divided into partitions for scalability
     - Topics have a configurable retention period (default: 7 days)
     - Topics can be compacted (keep only latest value per key)

   Topic Types:
     Regular    : Retain all events for a configured time
     Compacted  : Keep only the latest record per key (like a key-value store)


3. PARTITION
   ----------
   A topic is split into one or more partitions for parallelism and scalability.
   Each partition is an ordered, immutable sequence of records.

   Topic: "orders" with 3 partitions:
   в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
   в”Ӯ Partition 0: [msg0] [msg1] [msg4] [msg7] ...    в”Ӯ
   в”Ӯ Partition 1: [msg2] [msg5] [msg8] [msg11] ...   в”Ӯ
   в”Ӯ Partition 2: [msg3] [msg6] [msg9] [msg12] ...   в”Ӯ
   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

   Key facts about partitions:
     - Messages within a partition are strictly ordered
     - Ordering is NOT guaranteed across partitions
     - Each partition is stored on one broker (+ replicas on others)
     - More partitions = more parallelism = higher throughput
     - Partitioning strategy:
         * Key provided  вҶ’ hash(key) % num_partitions вҶ’ same key, same partition
         * No key        вҶ’ round-robin across partitions

   Offset:
     - Each message in a partition has a unique sequential offset (0, 1, 2, ...)
     - Consumers track their position using offsets
     - Offsets are per-partition (not global)


4. BROKER
   -------
   A Kafka broker is a single Kafka server.
   A Kafka cluster consists of multiple brokers working together.

   в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
   в”Ӯ              KAFKA CLUSTER                 в”Ӯ
   в”Ӯ                                            в”Ӯ
   в”Ӯ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ
   в”Ӯ  в”Ӯ Broker 1 в”Ӯ  в”Ӯ Broker 2 в”Ӯ  в”Ӯ Broker 3 в”Ӯ в”Ӯ
   в”Ӯ  в”Ӯ          в”Ӯ  в”Ӯ          в”Ӯ  в”Ӯ          в”Ӯ в”Ӯ
   в”Ӯ  в”Ӯ P0 (L)   в”Ӯ  в”Ӯ P1 (L)   в”Ӯ  в”Ӯ P2 (L)   в”Ӯ в”Ӯ
   в”Ӯ  в”Ӯ P1 (R)   в”Ӯ  в”Ӯ P2 (R)   в”Ӯ  в”Ӯ P0 (R)   в”Ӯ в”Ӯ
   в”Ӯ  в”Ӯ P2 (R)   в”Ӯ  в”Ӯ P0 (R)   в”Ӯ  в”Ӯ P1 (R)   в”Ӯ в”Ӯ

   в”Ӯ                                            в”Ӯ
   в”Ӯ  L = Leader partition                      в”Ӯ
   в”Ӯ  R = Replica partition                     в”Ӯ
   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

   - Producers write to the Leader partition of a broker
   - Consumers read from the Leader partition
   - Replicas are kept in sync on follower brokers
   - If a broker fails, a follower is elected as new Leader


5. PRODUCER
   ---------
   A producer is an application that writes (publishes) events to Kafka topics.

   Producer Flow:
   в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
   в”Ӯ  Application creates event                      в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Serialize (JSON / Avro / Protobuf)             в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Partitioner decides which partition            в”Ӯ
   в”Ӯ  (based on key hash or round-robin)             в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Batch messages in memory (for efficiency)      в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Send batch to Kafka broker (Leader)            в”Ӯ
   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

   Producer Acknowledgment Modes (acks):
     acks=0  : Fire and forget вҖ” no confirmation (fastest, may lose data)
     acks=1  : Leader acknowledges вҖ” moderate reliability
     acks=all: All ISR replicas acknowledge вҖ” strongest guarantee (slowest)

   Idempotent Producer:
     - Enable idempotence=true to prevent duplicate messages on retries
     - Kafka assigns a producer ID + sequence number to deduplicate


6. CONSUMER
   ---------
   A consumer is an application that reads (subscribes to) events from topics.

   Consumer Flow:
   в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
   в”Ӯ  Subscribe to topic(s)                          в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Poll for new messages                          в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Deserialize message                            в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Process message                                в”Ӯ
   в”Ӯ         вҶ“                                       в”Ӯ
   в”Ӯ  Commit offset (mark as processed)              в”Ӯ
   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

   Offset Commit Strategies:
     Auto commit    : Kafka commits offset periodically (may reprocess on failure)
     Manual commit  : Application commits offset after successful processing
                      (at-least-once or exactly-once delivery)

   Consumer reads from the beginning or from a specific offset:
     auto.offset.reset=earliest вҶ’ read all historical messages
     auto.offset.reset=latest   вҶ’ read only new messages

8. ZOOKEEPER / KRaft
   ------------------
   ZooKeeper (legacy):
     - Kafka used ZooKeeper for cluster coordination and metadata storage
     - Tracked: broker list, topic configs, partition leaders, consumer offsets
     - Required a separate ZooKeeper cluster to run Kafka

   KRaft (Kafka Raft вҖ” Kafka 3.3+ default):
     - Kafka's built-in consensus protocol replacing ZooKeeper
     - Kafka now manages its own metadata without external dependencies
     - Simpler deployment, faster recovery, supports 1M+ partitions
     - ZooKeeper is fully removed in Kafka 4.0+


9. REPLICATION
   -----------
   Kafka replicates each partition across multiple brokers for fault tolerance.

   Replication Factor:
     - replication.factor=3 вҶ’ 1 Leader + 2 Followers
     - Can lose up to (replication.factor - 1) brokers without data loss

   ISR (In-Sync Replicas):
     - Set of replicas that are fully caught up with the Leader
     - acks=all вҶ’ producer waits for all ISR to acknowledge
     - If a follower falls behind, it is removed from ISR

   Leader Election:
     - If Leader fails, Kafka elects a new Leader from ISR
     - Happens automatically within seconds


10. RETENTION POLICY
    -----------------
    Kafka retains messages based on configured policies:

    Time-based   : Delete messages older than N days (default: 7 days)
                   log.retention.hours=168

    Size-based   : Delete oldest messages when topic exceeds size limit
                   log.retention.bytes=1073741824  (1GB)

    Log Compaction: Keep only the LATEST record per key
                   Useful for: user profiles, inventory state, config changes
                   Deleted keys: tombstone record (null value) marks deletion


------------------------------------------------------------
KAFKA DELIVERY SEMANTICS
------------------------------------------------------------

At-Most-Once:
  - Message delivered 0 or 1 times
  - No retries вҖ” message may be lost
  - Use when: loss is acceptable (metrics, logs)

At-Least-Once:
  - Message delivered 1 or more times
  - May produce duplicates on retry
  - Consumer must be idempotent
  - Most common default

Exactly-Once:
  - Message delivered exactly once, end-to-end
  - Requires: idempotent producer + transactions + transactional consumer
  - Use when: financial transactions, inventory updates, billing
  - Slight performance overhead


------------------------------------------------------------
KAFKA STREAMS
------------------------------------------------------------

Kafka Streams is a Java library for building real-time stream processing
applications directly on top of Kafka. No separate cluster needed.

Key operations:
  - filter()     : Keep only events matching a condition
  - map()        : Transform each event
  - groupBy()    : Group events by key
  - aggregate()  : Accumulate values (count, sum, average)
  - join()       : Join two streams or a stream with a table
  - windowing()  : Process events in time windows (tumbling, sliding, session)

Example flow:
  Raw Clicks вҶ’ Filter (product pages only) вҶ’ Group by product_id
             вҶ’ Count per 5-minute window вҶ’ Publish to trending-products topic


------------------------------------------------------------
KAFKA CONNECT
------------------------------------------------------------

Kafka Connect is a framework to stream data between Kafka and external systems
without writing custom code.

  Source Connectors: Pull data FROM external systems INTO Kafka
    - MySQL вҶ’ Kafka (Debezium CDC connector)
    - PostgreSQL вҶ’ Kafka
    - S3 вҶ’ Kafka
    - REST API вҶ’ Kafka

  Sink Connectors: Push data FROM Kafka TO external systems
    - Kafka вҶ’ Elasticsearch
    - Kafka вҶ’ S3 / GCS
    - Kafka вҶ’ Snowflake / BigQuery
    - Kafka вҶ’ MongoDB

  Common connectors:
    Debezium     : CDC from MySQL, PostgreSQL, MongoDB, Oracle
    Confluent S3 : Stream Kafka data to Amazon S3
    JDBC         : Any JDBC-compatible database
    Elasticsearch: Stream to Elasticsearch index

PRODUCER CONFIGS:
  bootstrap.servers    : Kafka broker addresses
  key.serializer       : Serializer for message key (StringSerializer, etc.)
  value.serializer     : Serializer for message value
  acks                 : Acknowledgment level (0, 1, all)
  retries              : Number of retry attempts on failure
  batch.size           : Max bytes in a batch (default: 16KB)
  linger.ms            : Wait time before sending batch (improves throughput)
  compression.type     : none, gzip, snappy, lz4, zstd
  enable.idempotence   : true вҶ’ exactly-once delivery guarantee

CONSUMER CONFIGS:
  bootstrap.servers    : Kafka broker addresses
  group.id             : Consumer group name
  key.deserializer     : Deserializer for message key
  value.deserializer   : Deserializer for message value
  auto.offset.reset    : earliest | latest | none
  enable.auto.commit   : true (auto) | false (manual commit)
  auto.commit.interval.ms : How often to auto-commit (default: 5000ms)
  max.poll.records     : Max records returned per poll (default: 500)
  session.timeout.ms   : Time before consumer is considered dead

TOPIC CONFIGS:
  num.partitions       : Number of partitions (default: 1)
  replication.factor   : Number of replicas (recommended: 3)
  retention.ms         : Message retention time (default: 604800000 = 7 days)
  retention.bytes      : Max size per partition before deletion
  cleanup.policy       : delete | compact | compact,delete
  min.insync.replicas  : Min ISR to acknowledge writes (use with acks=all)


------------------------------------------------------------
KAFKA SERIALIZATION FORMATS
------------------------------------------------------------

String / JSON:
  - Simple, human-readable, no schema enforcement
  - Best for: development, simple pipelines

Apache Avro:
  - Binary format with schema registry
  - Compact, fast, schema evolution support
  - Best for: production pipelines with strong typing

Protocol Buffers (Protobuf):
  - Google's binary serialization format
  - Very compact and fast, language-neutral

Apache Parquet:
  - Columnar format, best for analytics (rarely used in Kafka directly)

Schema Registry (Confluent):
  - Central repository for Avro/Protobuf/JSON schemas
  - Ensures producers and consumers agree on schema
  - Supports schema evolution (backward, forward, full compatibility)

KAFKA USE CASES
------------------------------------------------------------

1. Real-Time Event Streaming
   - User clicks, page views, searches вҶ’ real-time analytics dashboard
   - Fraud detection: analyze payment events as they happen

2. Microservices Communication (Event-Driven Architecture)
   - OrderPlaced event вҶ’ inventory service, payment service, notification service
   - Decouples services вҖ” no direct API calls between services

3. Log Aggregation
   - Collect logs from hundreds of servers into Kafka
   - Forward to Elasticsearch, Splunk, or S3

4. Change Data Capture (CDC)
   - Debezium captures every database change (INSERT/UPDATE/DELETE)
   - Streams changes to downstream systems in real-time

5. Data Pipeline / ETL
   - Source systems вҶ’ Kafka вҶ’ Stream processing вҶ’ Data warehouse
   - Real-time ETL replacing nightly batch jobs

6. IoT Data Ingestion
   - Millions of IoT sensors publish readings to Kafka
   - Stream processing for anomaly detection, alerting

7. Activity Tracking
   - Track user behavior: views, clicks, purchases, searches
   - Feed into recommendation engine and personalization

------------------------------------------------------------
SUMMARY
------------------------------------------------------------

Apache Kafka is the backbone of modern real-time data architectures.

Core Concepts:
  вң” Event       : Immutable record with key, value, timestamp, offset
  вң” Topic       : Named channel вҖ” append-only, partitioned, retained
  вң” Partition   : Ordered log segment вҖ” enables parallelism and scaling
  вң” Broker      : Kafka server вҖ” stores partitions, serves producers/consumers
  вң” Producer    : Writes events to topics (acks: 0/1/all)
  вң” Consumer    : Reads events from topics, tracks position via offsets
  вң” Consumer Group: Parallel consumers sharing topic partitions
  вң” Replication : Leader + followers for fault tolerance
  вң” Retention   : Messages kept for days/weeks, replayable
  вң” KRaft       : Built-in consensus replacing ZooKeeper (Kafka 3.3+)

Ecosystem:
  вң” Kafka Streams  : In-process stream processing library
  вң” Kafka Connect  : Source/Sink connectors to external systems
  вң” Schema Registry: Central schema management for Avro/Protobuf

Why Kafka wins:
  - Handles millions of events per second
  - Messages are durable, replayable, and retained
  - Multiple consumers read the same data independently
  - Decouples producers from consumers (event-driven architecture)
  - Foundation of modern data pipelines, microservices, and real-time analytics
